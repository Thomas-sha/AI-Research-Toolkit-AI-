# NPS追踪体系搭建与分析

## 方法论核心

NPS（Net Promoter Score）不是一个数字，而是一个**客户体验的诊断系统**。绝大多数NPS项目止步于"今年NPS是XX分，比去年高了Y分"——这等于去医院体检只量了体温。

本Skill基于3年11+项目的NPS追踪实战经验，覆盖从指标体系设计到年度趋势追踪的完整闭环。核心思想：**NPS告诉你"有问题"，触点分析告诉你"哪里有问题"，结构方程告诉你"哪个问题最重要"，趋势追踪告诉你"问题在变好还是变坏"。**

### 四个核心原则

| 原则 | 说明 |
|------|------|
| **NPS≠终点** | NPS是体检温度计，不是诊断报告。价值在驱动因素分析，不在分数本身 |
| **触点即诊断点** | 将客户旅程拆解为可测量的具体触点，每个触点的体验评分 + NPS贡献权重 = 优化优先级 |
| **结构方程驱动** | 用SEM（结构方程模型）而非简单相关性来判断"哪个触点对NPS的贡献最大"。相关性≠因果，SEM更接近因果推断 |
| **趋势>单点** | 单次NPS的价值有限。年度追踪揭示的是"改善速度"和"竞争相对位置"，这才是决策层关心的 |

## 标准流程：5步 × AI加速

### Step 1：指标体系设计

**目标**：构建一个能回答"哪里出问题、哪个问题最重要"的多层指标体系。

#### 1.1 NPS核心指标层

```
Level 1：NPS总指标
  ├─ 净推荐值（NPS = %推荐者 - %贬损者）
  ├─ 推荐者比例 / 被动者比例 / 贬损者比例
  └─ 按平台/渠道/人群的NPS分解

Level 2：一级驱动因子（通常3-5个）
  ├─ 产品体验
  ├─ 购买体验
  ├─ 服务体验
  └─ 品牌感知

Level 3：二级触点指标（通常15-25个）
  ├─ 产品体验 → 产品质量/包装/使用效果/...
  ├─ 购买体验 → 页面体验/客服响应/物流速度/...
  ├─ 服务体验 → 售后响应/问题解决/主动关怀/...
  └─ 品牌感知 → 品牌信任/性价比感知/推荐意愿/...
```

**设计要点**：
- Level 2的数量取决于商业问题。如项目目标为"提升电商体验"，则购买体验下设更多触点
- Level 3的每个触点 = 问卷中的一个具体评分题
- 指标体系设计完成后，应与客户对齐——确认"这些维度覆盖了他们的管理范围，且他们有权对发现做改善"

#### 1.2 指标体系文档模板

```markdown
## NPS指标体系 · [项目名称/年份]

### NPS核心指标
| 指标 | 计算方式 | 数据来源 |
|------|------|------|
| 整体NPS | (推荐者数-贬损者数)/总样本数×100 | Q[NPS主问题] |
| 推荐者比例 | 评分9-10占比 | Q[NPS主问题] |
| 贬损者比例 | 评分0-6占比 | Q[NPS主问题] |

### 一级因子
| 因子 | 包含触点 | 对应问卷题号 |
|------|------|------|
| 产品体验 | 质量/包装/效果 | Q5-Q8 |
| 购买体验 | 页面/客服/物流 | Q9-Q14 |
| ... | ... | ... |

### 二级触点（完整清单）
| 触点ID | 触点名称 | 评估维度 | 评分方式 | 问卷题号 |
|------|------|------|------|:--:|
| T01 | 商品页面信息清晰度 | 信息完整/准确/易理解 | 1-10分 | Q9 |
```

### Step 2：问卷设计与DP Spec

**目标**：设计一份结构严谨、DP可高效处理、能支撑SEM建模的定量问卷。

#### 2.1 问卷结构

```
模块1：NPS主问题（1题）
  "您有多大可能向朋友/同事推荐[品牌/产品]？"（0-10分）

模块2：一级因子总体评价（3-5题）
  "请对以下方面的整体体验打分"（1-10分）

模块3：二级触点体验评分（15-25题）
  每个触点1题，使用统一标尺（1-10分）
  必须覆盖所有一级因子的触点

模块4：开放题（1-2题）
  "您给出这个评分的最主要原因是什么？"
  "您认为最需要改善的是什么？"

模块5：人口属性/行为标签
  根据配额需要（平台/会员等级/购买频次/城市级别）
```

#### 2.2 问卷设计原则

- **标尺统一**：全部使用1-10分，便于跨触点对比和SEM建模
- **触点互斥**：每个触点只测一个概念，避免"客服响应速度和态度"这种混在一起的问题
- **覆盖完整**：触点清单须覆盖客户旅程的全部关键环节。漏掉一个关键触点 = 分析出现盲区
- **配额前置**：样本条件和配额在问卷设计阶段就明确，不等到数据回收后再补充

#### 2.3 DP Spec 核心配置

DP（数据处理）spec是定量项目的"施工图"，决定了数据的切片方式和分析精度：

```markdown
## DP Spec · 核心配置

### Base（基础分组）
- 全部样本
- 按平台（京东/天猫/...）
- 按会员等级（普通/银牌/金牌/钻石）
- 按购买频次（首次/复购/高频）

### Header（表头变量）
- NPS分类（推荐者/被动者/贬损者）
- 入口渠道
- 城市级别

### 统计检验
- 显著性水平：95%置信区间
- 标注方式：字母标注法（ABCD）
```

### Step 3：数据回收与质控

**目标**：确保回收数据的质量足以支撑SEM建模和跨年度对比。

#### 3.1 QC规则设计

| 规则类型 | 示例 | 用途 |
|------|------|------|
| **答题时长** | 低于中位数50% → 标记 | 筛除"刷题"样本 |
| **直线作答** | 连续10题同分 → 标记 | 筛除敷衍样本 |
| **陷阱题** | "本题请选3" → 选3者保留 | 筛除不认真样本 |
| **开放题质量** | 无意义文字/乱码 → 标记 | 定性分析可用性保障 |
| **IP/设备去重** | 同设备多份 → 保留第一份 | 防止重复作答 |

#### 3.2 样本配额监控

- 数据回收期间每日监控配额完成率
- 软配额（性别/年龄/城市）允许±5%浮动
- 硬配额（平台/会员等级/购买时间窗口）不可浮动
- 配额偏差超过阈值 → 通知panel补充特定人群

### Step 4：数据分析——结构方程与回归

> 这是NPS项目的核心分析阶段，也是AI加速和人工判断最紧密结合的环节。

#### 4.1 分析框架

```
原始数据（SPSS/Excel）
  │
  ├─ 描述统计层
  │   ├─ NPS总分（整体 + 各base分组）
  │   ├─ 各触点平均分 & 分布
  │   └─ 推荐者/贬损者的触点评分差异
  │
  ├─ 驱动分析层（核心）
  │   ├─ 结构方程模型（SEM）：哪个一级因子→NPS的路径系数最大？
  │   ├─ 回归系数：哪个二级触点→该因子满意度的影响最大？
  │   └─ 触点权重 = SEM路径系数 × 回归系数
  │
  └─ 深度分析层
      ├─ 会员体验深度分析（按会员等级的差异化）
      ├─ 开放题文本分析（NPS评分原因的定性归纳）
      └─ 与去年数据的对比（趋势变化）
```

#### 4.2 结构方程模型（SEM）

**为什么用SEM而不是简单回归？**

NPS的驱动因素之间不是独立的。"物流快"会影响"整体购买体验"，"购买体验"又影响NPS——这是一条因果链。简单回归会把物流和购买体验对NPS的贡献重复计算，SEM通过潜变量建模，把重叠部分拆开。

**建模步骤**：

1. **模型设定**：根据指标体系设定测量模型（触点→因子）和结构模型（因子→NPS）
2. **模型拟合**：使用AMOS（或R的lavaan包），检查拟合指标（CFI>0.90, RMSEA<0.08）
3. **路径系数提取**：标准化路径系数 = 该因子对NPS的"净贡献"
4. **模型修正**：根据修正指数（MI）调整，删除不显著路径，增加被忽略的关键路径
5. **分群建模**：按平台/会员等级分别建模，识别不同人群的驱动结构差异

**输出示例**（脱敏）：

```
结构模型 · 标准化路径系数

产品体验 ──→ NPS    0.32***
购买体验 ──→ NPS    0.41***
服务体验 ──→ NPS    0.18**
品牌感知 ──→ NPS    0.09*

(*** p<0.001, ** p<0.01, * p<0.05)

→ 核心发现：购买体验是NPS的最强驱动因子（0.41），
  尤其对于高价值会员群体，路径系数高达0.52
```

#### 4.3 触点回归与权重计算

SEM告诉你"哪个因子最重要"，回归告诉你"在最重要的因子下，哪个触点最关键"。

**两步权重计算**：

```
触点重要性 = SEM路径系数(因子→NPS) × 回归系数(触点→因子)

示例：
产品体验→NPS的SEM系数 = 0.32
产品质量→产品体验的回归系数 = 0.45
→ 产品质量对NPS的综合权重 = 0.32 × 0.45 = 0.144

服务质量→产品体验的回归系数 = 0.30
→ 服务质量对NPS的综合权重 = 0.32 × 0.30 = 0.096
```

**输出格式**（脱敏）：

```markdown
## 触点重要性排名（基于SEM+回归综合权重）

| 排名 | 触点 | 所属因子 | 综合权重 | 当前均分 | 改善潜力 |
|:--:|------|------|:--:|:--:|:--:|
| 1 | 物流配送速度 | 购买体验 | 0.18 | 7.2 | 🔴 高权重-低满意度 |
| 2 | 客服问题解决率 | 购买体验 | 0.15 | 8.1 | 🟡 高权重-中满意度 |
| 3 | 产品使用效果 | 产品体验 | 0.14 | 8.9 | 🟢 高权重-高满意度 |
| ... | ... | ... | ... | ... | ... |
```

**改善优先级矩阵**：

```
        权重
          ↑
    高    │  🔴 优先改善区     🟢 保持优势区
         │  (高权重-低满意)    (高权重-高满意)
         │
    低    │  ⚪ 低优先级区      🟡 观察区
         │  (低权重-低满意)    (低权重-高满意)
         └──────────────────────→ 满意度
              低              高
```

#### 4.4 开放题分析

将NPS开放题（"您给出这个评分的最主要原因是什么？"）按以下维度编码：

| 维度 | 编码规则 | 用途 |
|------|------|------|
| 提及率 | 某触点在所有开放题中被提及的频次 | 补充定量——定量测"满意度"，开放题暴露"心智显著性" |
| 情感倾向 | 正面/中性/负面 | 区分推荐者和贬损者的焦点差异 |
| 具体场景 | 用户描述的具体使用/购买场景 | 对定量数据做"为什么"的解释 |

### Step 5：Dashboard与趋势追踪

**目标**：将分析结果固化为可年度复用的Dashboard，支持跨年趋势对比。

#### 5.1 Dashboard设计原则

```
Dashboard页面结构（建议6-8页）：

Page 1：NPS总览
  - NPS总分（当前期 vs 上期 vs 去年同期）
  - 推荐者/被动者/贬损者比例变化
  - 按平台/会员等级的NPS分解

Page 2：一级因子得分趋势
  - 各因子当前得分 & 年度变化
  - 各因子对NPS的贡献权重变化

Page 3：触点重要性矩阵（改善优先级）
  - 权重 × 满意度气泡图
  - 标注top 5改善优先级触点

Page 4：触点得分热力图
  - 所有触点在所有base分组下的得分
  - 红-黄-绿色阶，一眼定位问题区域

Page 5：会员分层分析
  - 按会员等级的NPS差异
  - 各等级会员的驱动因子差异

Page 6：开放题词云与关键引述
  - 推荐者高频词 vs 贬损者高频词
  - 代表性用户原声

Page 7：年度趋势对比（如有历史数据）
  - NPS年度变化曲线
  - 触点得分的年度变化（标注显著改善/恶化项）

Page 8：行动建议
  - 优先改善的3-5个触点
  - 对应的改善行动建议和预期NPS提升
```

#### 5.2 年度追踪的"一致性"原则

跨年度NPS追踪最容易被忽视的挑战：**问卷变了，数据就不具可比性**。

| 必须保持一致 | 允许调整 |
|------|------|
| NPS主问题的措辞和标尺 | 新增触点（新业务/新功能上线） |
| 核心触点的定义和评分方式 | 删除不再相关的触点 |
| 样本条件和配额框架 | 配额比例根据业务重心微调 |
| SEM模型的核心结构 | 新增路径（新触点→已有因子） |

> 实操规则：每年问卷调整后，做一份"变更对照表"，标注哪些指标与上年直接可比、哪些需要注明年份差异。

#### 5.3 趋势分析的三个核心问题

| 问题 | 分析方法 | 决策意义 |
|------|------|------|
| NPS变了——是真变化还是样本波动？ | 统计显著性检验（t-test跨年） | 避免因随机波动做出错误决策 |
| 哪些触点的改善/恶化在驱动NPS变化？ | 触点得分的年度变化 × 该触点的SEM权重 | 定位NPS变化的具体来源 |
| 驱动结构是否在变化？ | 今年SEM模型 vs 去年SEM模型——路径系数是否有显著变化？ | 发现用户体验优先级的结构性转变 |

|------|------|

## 人工判断的四个不可替代节点

### 1. 🔴 指标体系与商业问题的对齐

AI能帮你列出50个可能的触点，但**哪些触点应该纳入、哪些应该舍弃、每个触点测量到什么深度**——取决于你对客户组织架构和管理范围的理解。一个触点如果客户内部没有团队能对它做改善，测了也是白测。

### 2. 🔴 SEM模型的修正决策

AI可以跑出模型拟合结果和修正指数（MI），但**是否接受一个MI建议的路径、是否删除一个不显著的路径**——这是方法论判断：
- 新增一个路径在统计上有意义，在逻辑上是否合理？
- 删除一个不显著的路径，是否因为这个因子确实不重要，还是因为样本量不够？
- 分群后路径系数差异大，是真实的群体差异还是模型不稳定？

### 3. 🔴 NPS变化的"信号 vs 噪音"判断

AI会告诉你"NPS从78降到73，变化5分"，但你的判断在于：
- 这个降幅在行业惯例中算大还是算小？
- 5分的下降中，有多少是样本结构变化（今年多了XX%的新用户）造成的？
- 这是需要紧急行动的红灯，还是"再观察一期"的黄灯？

### 4. 🔴 行动建议的优先级与现实感

> "AI可以列出'需要改善的top 5触点'，但它不知道客户内部哪个部门有预算、哪个部门正在推动变革、哪个部门的VP是项目sponsor。"

最佳的行动建议不是"科学上最优"的那个，而是**客户组织内部能落地、有ownership、且能在下一期追踪中看到效果**的那个。

### 案例1：某国际润滑油品牌 · 电商NPS 3年追踪

| 维度 | 详情 |
|------|------|
| **项目周期** | 2022-2025，每年2期（上半年/下半年），累计6期以上追踪 |
| **覆盖平台** | 京东 + 天猫双平台独立追踪 |
| **指标体系** | NPS总指标 + 4个一级因子（产品/购买/服务/品牌）+ 20+个二级触点 |
| **核心分析方法** | 结构方程模型（AMOS）+ 触点回归系数 + 年度趋势对比 + 会员分层分析 |
| **关键发现类型** | 购买体验（物流+客服）是NPS最强驱动因子，权重超过产品体验；高价值会员的驱动结构与普通用户显著不同 |
| **Dashboard** | 6-8页标准化Dashboard，每期更新，支持跨年趋势对比 |
| **AI加速亮点** | SPSS syntax自动生成、SEM模型辅助设定、开放题批量编码、年度趋势变化的自然语言解读初稿 |
| **方法论演进** | 首年（2022）：定性深访（场景+触点梳理）+ 定量验证 → 次年（2023-2025）：定量追踪 + 触点拓展 + 可视化平台 |

## 常见陷阱与对策

| # | 陷阱 | 表现 | 对策 |
|:--:|------|------|------|
| 1 | **只报NPS分数不做驱动分析** | 报告正文大量篇幅讨论"NPS上升/下降"本身 | NPS是体检温度计。在报告第一页放分数，剩下20页讲驱动因素、触点权重、改善优先级 |
| 2 | **用简单相关性替代SEM** | "物流满意度与NPS的相关系数为0.45，所以物流最重要" | 相关性≠因果链。物流满意度可能通过"购买体验"间接影响NPS。用SEM拆开直接效应和间接效应 |
| 3 | **问卷改了但没做可比性标注** | 今年新增了5个触点，但趋势图直接把今年和去年放一起 | 做"变更对照表"，标注哪些新增、哪些删除、哪些措辞有改动。不可比的数据不要强行比 |
| 4 | **忽视样本结构变化** | 今年NPS上升，其实是因为高价值会员样本占比从30%变到45% | 按base分组分别看NPS趋势。如果整体NPS变了但各组都没变——是样本结构变了，不是体验变了 |
| 5 | **SEM模型过度修正** | 根据MI把所有"统计上显著"的路径都加上，模型变成了没有理论意义的path spaghetti | SEM是理论驱动的。每个新增路径必须有逻辑解释。修正指数是建议，不是命令 |
| 6 | **行动建议不可落地** | "改善物流速度"——客户物流部门说"这归第三方物流管" | 每个建议在写之前先问：客户内部哪个团队能做这件事？他们有没有预算和KPI？ |
| 7 | **年度追踪只看NPS不看驱动结构** | 连续3年追踪，只做了NPS分数对比，没发现"驱动结构已悄然改变" | 每年至少跑一次SEM并与历史对比。用户对"什么是好体验"的期待本身在变 |

## 与其它Skill的联动

- **`desk-research-ai`** → 行业趋势扫描 + NPS数据 = 行业标杆对比（"我们的NPS在这个行业处于什么位置？"）
- **`cross-tab-analyst`** → NPS按base分组的交叉分析可由交叉表Skill接管
- **概念/产品测试**（规划中）→ 新品上市前的NPS预评估 → 上市后的NPS追踪 = 完整的产品体验闭环
- **`方法论-未来需求研究完整流程`** → 未来需求研究中发现的用户场景 → NPS追踪中验证这些场景的体验满意度
